# Claude Code Project Context

This file provides context for Claude Code to align its behavior with the project's Spec-Driven Development (SDD) workflow and Antigravity agents.

## Agent Personas & Roles
This project uses a multi-agent orchestration strategy. Persona definitions are stored in `.antigravity/agents/`. When asked to "Switch to [Agent Name]", read the corresponding file in that directory to adopt its rules and principles.

- **Architect**: `.antigravity/agents/architect.md`

- **Backend Engineer**: `.antigravity/agents/backend_engineer.md`

## Build & Test Commands

### iOS (Native)
- **Build**: `xcodebuild -workspace ios-app/*.xcworkspace -scheme YourScheme build` (Adjust for your scheme name)
- **Test**: `xcodebuild -workspace ios-app/*.xcworkspace -scheme YourScheme test -destination 'platform=iOS Simulator,name=iPhone 15'`
- **Lint**: `swiftlint` (if installed)

### Backend (Vercel/Next.js)
- **Install**: `npm install`
- **Dev**: `npm run dev` or `npx vercel dev`
- **Test**: `npm test`
- **Build**: `npm run build`

## Project Structure & Source of Truth
- **Source of Truth**: All implementation must follow the specifications in the `specs/` directory.
- **Artifacts**: Check `artifacts/` for implementation plans and walkthroughs generated by other agents.
- **Safety**: Do NOT modify `.xcodeproj` or `.pbxproj` files directly.

## Engineering Philosophy
- **Simple interfaces**: Design clear module and API boundaries. Depend on the public contract and observable behavior of the other side, not on how it happens to be implemented.
- **Performance by design**: Prefer good data flow, state shape, boundaries, and I/O choices over clever implementation tricks. Optimize internals only after the design is sound and evidence says it matters.
- **Single responsibility**: Keep units small and focused. If a module has multiple reasons to change, split the responsibility before adding more behavior.
- **Explicit state machines**: Prefer named states and valid transitions over scattered booleans or implied state. Scope state variables to the state that owns them so invalid states are hard or impossible to represent.
- **Behavior-based tests**: Favor fast unit tests that exercise public behavior through the public interface. If behavior is hard to test without reaching into internals, refactor the interface or design.
- **Use judgment**: Apply these principles in the idiom of the language, framework, and platform. Simplicity should reduce friction, not fight the tool.

## Coding Style
- **iOS**: SwiftUI, async/await, Observation framework, SwiftData for persistence.
- **Backend**: TypeScript, Next.js API routes, Type-safe ORMs (Prisma/Drizzle).
- **Communication**: Be concise. Use Markdown for all documentation.
